## §3 tools 계약 (게이트 실측 — Swift 구현 규격, 수술 B1)

**요청** (MessagesRequest 확장):
```json
"tools": [{ "name": "web_search", "description": "...", "input_schema": { "type":"object", "properties": {"query": {"type":"string"}}, "required":["query"] } }]
```

**비스트림 응답**: `content[].type == "tool_use"` → `{id, name, input(object)}`, `stop_reason == "tool_use"`. MessagesResponse 디코더에 블록 분기 추가.

**스트림 SSE 순서** (실측):
```
content_block_start  {index, content_block:{type:"tool_use", id, name}}
content_block_delta  {index, delta:{type:"input_json_delta", partial_json:"..."}}   ← 단편 누적
content_block_stop   {index}                                                        ← JSON 완성 시점
message_delta        {delta:{stop_reason:"tool_use"}}
message_stop
```
- `StreamChunk` 에 `content_block`(type·id·name)·`delta.partial_json` 필드 추가. 클라이언트는 partial_json 누적 버퍼 유지.
- `StreamEvent` 에 `.toolUse(id:String, name:String, inputJSON:String)` 추가 — `content_block_stop` 시점(tool_use 블록일 때) 방출.
- text 블록과 혼재 방어(텍스트 후 tool_use). 기존 `.delta` 경로 불변(회귀 0 원칙).
- `input` 은 object(실측). inputJSON 파싱 실패 시 해당 호출 무시 후 `.done` (방어).

**wire 모델 확장(수술 c)**: tool 왕복 회신을 위해 wire 메시지 content 가 `String | [블록]` 양형을 인코딩할 수 있어야 한다 — `WireMessage.content` 를 enum(.text(String) / .blocks([WireBlock]))으로 확장하거나 병렬 typed 경로 추가. **UI 의 ChatTurn 은 String 유지**(표시용), tool 왕복은 §4 의 라운드 내부 휘발 컨텍스트로만 존재.

**tool_result 회신**: messages 에 `{"role":"assistant","content":[{type:"tool_use",...}]}` + `{"role":"user","content":[{"type":"tool_result","tool_use_id":"...","content":"<검색결과 텍스트>"}]}` 추가 후 재요청(게이트 #2 실측 형식).

## §4 ChatViewModel tool 루프 (수술 B2 — 엄격 한도)

```
send():
  wireContext = 히스토리(텍스트 턴만, 기존 슬라이딩 윈도우) + 신규 user 턴
  for round in 0..<3 {                                  // 최대 진입 3회 = tool 2라운드 + 최종
      tools = (웹검색가능 && round < 2) ? [webSearch] : nil   // 3회째는 tools 제거→강제 최종 답변
      stream(wireContext, tools) 소비:
        ├─ .delta(text)  → 어시스턴트 버블 스트리밍(기존 동일)
        ├─ .toolUse(id,name,input) →
        │     state = .searching(query)
        │     중복: 이번 send 내 동일 query 재요청이면 결과 재사용(재검색 안 함)
        │     result = TavilySearchService.search(query)   // 실패 → tool_result 에 오류 문구
        │     wireContext += [assistant(tool_use)] + [user(tool_result)]
        │     continue round                              // 재스트리밍
        └─ .done → break
  }
  히스토리 반영: user 질문 + 최종 assistant 텍스트만 저장.
  ※ tool_use/tool_result 턴은 라운드 내부에서만 사용하고 영속 히스토리에서 제외
    → 슬라이딩 윈도우가 tool 쌍을 절단(짝 깨짐 API 오류)할 가능성을 구조적으로 차단.
취소: 검색 중에도 task.cancel() 즉시 전파(Tavily await 도 같은 Task 내 — 기존 §8.5 패턴 유지).
웹검색 가능 조건: KeychainStore.tavilyKey 비어 있지 않음.
```
- **tool_result 포맷**: `[1] 제목 — 발췌(≤400자) (URL)` 번호 목록, 최대 5건, 총 ≤3,000자.
- **system prompt 추가 지침**: "최신 정보·시세·뉴스·날씨 등 실시간성 질문에만 web_search 사용. 검색 결과로 답할 때 출처를 마크다운 링크로 표기."
- **프로필 주입**: userName/userIntro 비면 생략, 있으면 system 끝에 `\n\n[사용자 정보] 이름: …, 소개: …`.

## §5 브리핑 흐름 (BriefingViewModel 상태머신 + 세션 격리 M2)

```
idle → fetchingWeather → generating(스트리밍) → done(텍스트+날씨 스냅샷)
                ↓ 실패                    ↓ 실패
        failed(.weather: 캐시+재시도)  failed(.llm: 날씨 원자료 카드+재시도)
```
- **트리거**: ① 알림 탭(라우팅 M1) ② 하루 첫 진입(briefingEnabled & 오늘 캐시 없음) ③ 툴바 ☀️ 수동. ②③은 시트로 표시.
- **요청**: `/v1/messages` 1턴 — user 메시지에 날씨 요약(코드북 한국어 변환)+날짜·요일+도시+프로필. system "아침 브리핑 비서: 2~4문장, 우산/옷차림 등 실용 조언, 마크다운". **tools 미포함**(단방향 생성). 기존 ChatViewModel 과 무관한 독립 클라이언트 호출(세션 격리).
- **캐시**: UserDefaults `briefing.cache` = {date(yyyy-MM-dd), text, weatherSnapshot}. 당일 재진입 시 캐시 표시 + 새로고침 버튼(재생성).
- **이어서 대화**: `ChatViewModel.seedFromBriefing(text)` — **newChat() 후** `user("오늘 아침 브리핑 해줘")+assistant(브리핑)` 2턴 시드(Anthropic 첫 턴 user 규칙 준수) → 시트 닫기. 기존 대화가 있으면 확인 다이얼로그("새 대화로 시작됩니다").
- **알림 라우팅(M1)**: `AppDelegate(UNUserNotificationCenterDelegate)` — `didReceive`(탭)에서 `NotificationRouter.shared.pendingBriefing = true` → **ChatView** 감지(onAppear 선소비+`.onChange`) → BriefingView 시트. (QA-r2 승인: RootView 대신 ChatView — seedFromBriefing 배선 응집도) `willPresent`(앱 foreground 중 발화)는 `.banner` 표시. 알림 등록은 설정 토글 on 시 + 앱 시작 시 재등록(7일 재서명 대응).

## §6 날씨 데이터 (Open-Meteo)

- 요청: `https://api.open-meteo.com/v1/forecast?latitude&longitude&current=temperature_2m,apparent_temperature,precipitation,weather_code,wind_speed_10m,relative_humidity_2m&daily=temperature_2m_max,temperature_2m_min,precipitation_probability_max,weather_code&timezone=Asia/Seoul&forecast_days=1` (실측 OK)
- WMO weather_code 코드북: 0 맑음 / 1-3 구름 / 45-48 안개 / 51-67 이슬비·비 / 71-77 눈 / 80-82 소나기 / 95-99 뇌우 → (한국어 설명, SF Symbol).
- `WeatherServiceProtocol { func today(for: KoreanCity) async throws -> WeatherToday }` — Open-Meteo 구현체 교체 가능.

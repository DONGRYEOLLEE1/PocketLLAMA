**1. 총평 (3~5줄)**

plans/v0.2-memory-enhancement-plan.md은 연구 문서의 패턴을 나열했으나, 실제 코드베이스의 구체적 제약(단일 --jinja 35B 인스턴스, historyWindow=12 + maxToolRounds=3 하드캡 tool 루프, ToolDefinition의 query-only 고정 스키마, ScenePhase 완전 부재, UserDefaults 수준 영속만 존재)과 정면으로 충돌한다. iOS 백그라운드 실행 한계·FTS5 external content 동기화 불가·기존 tool 루프 예산 경쟁을 "게이트로 나중에" 미루는 구조라 구현 착수 시 즉시 깨진다. 이전 gemini 엄중 리뷰가 동일 Blocker를 이미 지적했는데도 본 초안은 수정 없이 그대로 남아 있다. YAGNI 과설계(1인 사용자 대상 LLM 모호판정·감쇠·bi-temporal)와 게이트/테스트 미비가 결합해, §9 DoD 통과 가능성이 낮다. 착수 전 전면 재설계 필요.

**2. Blocker**

- **plans/v0.2-memory-enhancement-plan.md:16 (M-D5), §2 (PocketLlamaApp.swift 변경), §3 (트리거: newChat/ScenePhase.background)** [Blocker]
  iOS ScenePhase.background에서 "수십초 LLM 추출 배치(추출+임베딩+SQLite 쓰기)"를 돌리는 설계.
  **증거**: PocketLlamaApp.swift:70-75는 WindowGroup + RootView만, @Environment scenePhase·onChange·UIApplicationDelegate scene 메서드 전무. ChatViewModel.swift:252-259 newChat()은 cancel+clear+store.clearSession()뿐, 어떤 배치도 없음. grep 전체 app/PocketLlama에서 ScenePhase / beginBackgroundTask / UIBackgroundModes 문자열 0건. iOS beginBackgroundTask 보증은 ~30초 내외이며 SIGKILL 시 트랜잭션 중단으로 DB 잠금/부분 메모리 발생.
  **1줄 대안**: 배치 추출을 background 즉시 실행에서 폐기하고, newChat()/send() 경계에서 "미처리 펜딩"만 빠르게 로컬 기록 → 포그라운드 복귀/idle 또는 다음 newChat 시점에만 실제 LLM 호출.

- **plans/v0.2-memory-enhancement-plan.md:44 (§1 DDL: CREATE VIRTUAL TABLE memory_fts USING fts5(text, content='memory'))** [Blocker]
  external content table 선언만 있고 동기화 트리거가 전혀 없음.
  **증거**: 계획서 어디에도 INSERT/UPDATE/DELETE AFTER TRIGGER DDL이 없음. SQLite FTS5 external content는 원본 테이블 변경을 자동 반영하지 않음(표준 동작). 또한 PK가 TEXT UUID(id)라 rowid(INTEGER) 기반 FTS5 매핑이 UPDATE/DELETE 시 깨지기 쉽다.
  **1줄 대안**: DDL에 rowid INTEGER PRIMARY KEY + 3종 AFTER TRIGGER(INSERT/UPDATE/DELETE 시 memory_fts(rowid,text) 동기화)를 명시하거나, contentless FTS5 + 수동 관리로 전환.

- **app/PocketLlama/ViewModels/ChatViewModel.swift:24,129-131 + Models/MessagesRequest.swift:94-126,216-220 (maxToolRounds=3 + ToolDefinition query-only)** [Blocker]
  save_memory/search_memory를 "기존 T2 tool 루프에 추가"(M-D11, §6 M4)하면 web_search와 라운드 예산(0·1 라운드만 tools 허용) 경쟁 + 과호출.
  **증거**: ChatViewModel.swift:131 `let tools: [ToolDefinition]? = (webEnabled && round < maxToolRounds - 1) ? [.webSearch] : nil`. MessagesRequest.swift:117 `static let webSearch`만 존재 + InputSchema는 properties.query 단일 고정. queryFromInput은 obj["query"]만 파싱. Wire blocks(toolUse/toolResult)는 이미 지원되지만 ToolDefinition 자체가 일반화되어 있지 않음.
  **1줄 대안**: ToolDefinition을 name+description+any JSON schema로 일반화하고, memory tools는 chat tool 루프 예산에서 분리(또는 별도 경량 path)하거나 maxToolRounds를 명시적 예산 모델로 교체.

**3. Major**

- **§1, §8 (FTS5 unicode61 + 브루트포스 코사인 + 가중치 미정)** [Major]
  한국어 조사·형태소 한계 미해결 + 매 send()마다 전체 행 브루트포스 코사인 비용 + α/β/γ·θ 초기값 부재.
  **1줄 대안**: FTS5 생성 시 trigram 명시 + 점수 가중치와 last_accessed 갱신 빈도(세션당 1회 상한)를 §8에 구체값으로 고정.

- **§3, M-D6 (35B-A3B 추출 지속성/부정문 안정성)** [Major]
  1-shot 프롬프트 + 빈 배열 폐기로 안정적이라는 가정. 미묘한 부정문·지속성 필터 오류가 조용히 쌓임.
  **1줄 대안**: 추출 프롬프트를 "지속성/부정문" 전용 few-shot + self-consistency로 강화하거나, 초기 버전은 단순 append-only + 수동 편집으로 축소.

- **M-D7/M-D12 + 프라이버시 언급 부재** [Major]
  soft-delete(valid_to)만으로 "삭제"를 대체하고, iOS Data Protection vs DB 암호화, 물리적 소거를 명시하지 않음.
  **1줄 대안**: 파일 보호(NSFileProtectionCompleteUntilFirstUserAuthentication 이상) + "진짜 삭제" 시 DELETE + VACUUM을 UI에서 지원.

- **§5, §7 (게이트/테스트 미비)** [Major]
  background partial write recovery, FTS5 트리거 정합성, 3-round tool 예산 경쟁, 한국어 부정문 추출, embedding dim 변경 마이그레이션, last_accessed write 증폭, 삭제 감사가 없음.
  **1줄 대안**: M0에 임베딩 인스턴스+계약 실측+serve.sh 이중 기동을 server-gate 스타일 자동 스모크로, §7에 위 항목을 구체 통과 기준으로 추가.

- **M-D9/M-D11 + §2 (YAGNI 과설계)** [Major]
  1인 사용자·폰 로컬에서 LLM ambiguity judge, 지수감쇠, 프루닝, bi-temporal, importance 1-10, verified 배지, 전용 MemoryViewModel까지 한 번에.
  **1줄 대안**: M1을 core 프로필 구조화로 한정, 장기 기억(M3)은 append-only + recency top-K + 수동 편집부터 시작.

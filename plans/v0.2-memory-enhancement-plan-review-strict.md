## ① 총평 + Go/No-Go

**판정: No-Go (현 초안 그대로 착수 금지) → 재설계 후 조건부 Go.**

이 계획서는 리서치(`research-personalized-agent-memory.md`)의 일반 아키텍처를 PocketLlama에 맞춰 구체화하려는 시도로서 방향(3계층·온디바이스 SQLite·세션 배치 추출·HITL·프라이버시 우선)은 옳다. 그러나 **현 코드베이스의 실제 제약과 정면으로 충돌하는 Blocker가 3건** 있고, 세 리뷰어(내부·Gemini·Grok)가 **독립적으로 동일 Blocker 3건에 수렴**했다. 핵심은 "게이트로 나중에" 미룰 수 없는, 설계 단계에서 지금 고쳐야 할 결함들이다:

1. **ScenePhase.background 추출 트리거가 iOS 백그라운드 실행 모델과 맞지 않는다** — 수십초 LLM 배치를 백그라운드에서 보장 실행할 수 없다. 데이터 정합성 리스크.
2. **FTS5 external content 테이블에 동기화 트리거가 없어 키워드 폴백이 작동조차 하지 않는다** — 내가 직접 SQLite로 재현 확인(매치 0건).
3. **save/search_memory를 기존 tool 루프에 넣는 설계가, query-only로 고정된 `ToolDefinition` 스키마와 3라운드 하드캡과 충돌한다.**

여기에 한국어 FTS5 한계(완화책도 불완전), 추출 품질 근거 부재, 프라이버시 미명시(암호화·진짜 삭제), YAGNI 과설계(1인 사용자에 bi-temporal·LLM 모호판정·감쇠·프루닝 일괄 도입), 게이트/테스트 미비가 Major로 겹친다.

**다만 이는 폐기가 아니라 다운스코프·재설계로 충분히 구제된다.** 아래 ④ 권고대로 (a) 추출을 포그라운드/유휴/newChat 경계로 옮기고, (b) FTS5 트리거를 명시하거나 더 단순한 `LIKE` 폴백으로 바꾸고, (c) 메모리 tool 경로를 분리(또는 M-D9 암묵 주입으로 일원화)하고, (d) v0.2 1단계를 "core 프로필 + append-only 장기기억 + 수동 편집"으로 좁히면 안전한 Go가 된다.

## ③ 사실성 검증 (맞음 / 틀림 / 미확인)

내부 리뷰에서 **SQLite 3.51 로 직접 재현 실측**한 항목을 포함한다(추측 아님).

### A. iOS 백그라운드 추출(ScenePhase.background) — **사실: 설계 위험 맞음**
- iOS에서 앱이 백그라운드 전환 시 `beginBackgroundTask`로 얻는 보장 실행 시간은 통상 **~30초 내외**(엄밀히는 OS 재량, 과거 ~3분에서 축소). 35B-A3B 원격 추론(TTFT만 수 초~수십 초, 코드의 `timeoutIntervalForRequest = 180` 이 35B 지연을 이미 인정) + 임베딩 왕복 + SQLite 쓰기를 background에서 완주할 보장이 없다. 미완료 SIGKILL 시 부분 쓰기/트랜잭션 중단 가능.
- **현 코드 사실(실측)**: `PocketLlamaApp.swift`에 `scenePhase`/`@Environment(\.scenePhase)`/`onChange`/`beginBackgroundTask`/`UIBackgroundModes` **전무**(grep 0건). §2의 "PocketLlamaApp.swift [변경] ScenePhase.background → 추출 배치 트리거"는 **존재하지 않는 인프라를 전제**한 신규 작업이며, 그마저도 위 제약과 충돌.
- **리서치 자체가 경고**: `research-personalized-agent-overview.md:95` "iOS 백그라운드 제약으로 상시 센싱·선제 폴링은 불가". 본 계획서가 자기 리서치 근거와 충돌.
- **판정: ScenePhase.background 즉시 추출은 폐기. newChat 경계 + 포그라운드 유휴 처리로.** (3시각 만장일치)

### B. FTS5 external content 동기화 — **사실: Blocker 맞음 (실측 재현)**
- 계획서 §1의 정확한 DDL(`CREATE VIRTUAL TABLE memory_fts USING fts5(text, content='memory')`)로 테이블 생성 + INSERT 후, **트리거 없이는 `memory_fts MATCH`가 0건** 반환(저장한 정확한 토큰조차 미매치)을 SQLite 3.51로 재현 확인. external content는 표준상 자동 동기화하지 않음 — **AFTER INSERT/UPDATE/DELETE 트리거 3종 필수**.
- **판정: 사실. DDL에 트리거 명시 또는 contentless/일반 FTS5로 전환.** (3시각 + 실측 일치)

### C. FTS5 한국어(unicode61) + trigram 완화책 — **사실이지만 완화책도 불완전 (실측으로 정밀화)**
- **unicode61**: "매운 음식을 못 먹는다" 저장 후 root "음식" 검색 → **0건**, 조사 붙은 "음식을" → 1건. 한국어 교착어 조사 결합 때문에 root-noun 회상 실패. Gemini/Grok 지적 맞음.
- **trigram 완화책(Gemini/Grok 권고)은 부분적으로만 맞다(실측 정정)**: trigram은 **3자 이상 쿼리만** 매치. "음식을"(3자) → 1건이지만 **bare "음식"(2자) → 0건**. 즉 짧은 2자 한국어 root는 trigram으로도 실패.
- **더 단순·견고한 대안(실측)**: 계획서 규모(수백 건)에선 인덱스 없는 **`text LIKE '%query%'` 전체 스캔이 2자 부분일치까지 정확히 동작**(실측 1건)하고 트리거·토크나이저 문제를 모두 회피. FTS5의 BM25 랭킹이 굳이 필요 없는 폴백 용도라면 LIKE가 YAGNI에 더 부합.
- **판정: FTS5 한국어 한계 = 사실. trigram = 부분 완화(불완전). v0.2 폴백은 LIKE 스캔 또는 trigram+최소길이 가드 중 택1.**

### D. `id TEXT PRIMARY KEY` ↔ FTS5 rowid — **Gemini 주장 과장 (실측 정정)**
- Gemini는 이를 Major("UUID PK가 rowid 매핑을 깨 극도로 난해")로 올렸으나, **실측 결과 `TEXT PRIMARY KEY` 테이블도 정수 `rowid`(1,2,…)를 그대로 보유**(`WITHOUT ROWID`가 아닌 한). 트리거를 `new.rowid` 기준으로 걸면 external content FTS5가 정상 작동함을 실측 확인.
- Grok은 같은 점을 "깨지기 쉽다"로 더 약하게 표현 — 더 정확.
- **판정: Blocker/Major 아님. 실제 caveat은 "트리거가 `new.id`가 아니라 `new.rowid`를 참조해야 하고, 앱이 FTS 히트를 rowid로 역매핑해야 한다"는 구현 주의(Minor).** Gemini 라벨 기각·하향.

### E. ToolDefinition 스키마 / tool 루프 — **사실 맞음 (실측)**
- `MessagesRequest.swift:94-126`: `ToolDefinition.InputSchema.Properties`는 **`query` 단일 필드로 하드코딩**, `static let webSearch`만 존재, `queryFromInput`은 `"query"` 키만 파싱. → `save_memory(text,type,importance)` 다중 인자 tool은 **현 구조로 인코딩 불가** → 스키마 일반화(`[String: Property]` 또는 임의 JSON) 필요. §2의 "AnthropicChatClient.swift [변경 없음 — tool 정의만 추가]"는 **변경 규모를 과소평가**.
- `ChatViewModel.swift:24,131`: `maxToolRounds=3`, tools는 `round < 2`에서만 주입(마지막 라운드 강제 무tools). search_memory(1R)+web_search(1R)면 예산 소진 → 과호출·누락 위험.
- **양호한 사실**: `MessagesRequest.Wire`는 이미 `.blocks([.toolUse/.toolResult])` 지원(실측) — 메모리 tool 왕복 wire는 재사용 가능.
- **판정: tool 결함 = 사실. 스키마 일반화 + 메모리 경로 분리.** (3시각 일치)

### F. serve.sh 임베딩 인스턴스 — **사실: 미구현 (실측)**
- `server/serve.sh`는 `--jinja` 단일 인스턴스(포트 8080)만 기동. `--embeddings`/`--pooling`/포트 분리/두 번째 llama-server **없음**. M-D13/§8의 "이중 기동 옵션 추가"는 미작성 작업 — **M0 게이트의 선결 산출물로 명시 필요.**

### G. SQLite FTS5 가용성(iOS) — **사실: 가용 확인**
- iOS 내장 SQLite는 FTS5 컴파일 포함(Apple 플랫폼 기본). `import SQLite3` raw 경로로 .xcodeproj 수정 없이 사용 가능. 계획서 전제 맞음. (단 C의 토크나이저 한계는 별개.)

## ④ 수정 권고 (계획서 반영용 — 우선순위)

**[P0 — Blocker, 착수 전 필수]**
1. **추출 트리거 재설계(M-D5/§2/§3)**: `ScenePhase.background 즉시 추출`을 폐기. 대화 종료 시 **원시 transcript를 "pending" 큐로 즉시 기록(빠른 쓰기)** → 실제 LLM 추출·임베딩은 **(a) 다음 `newChat()` 시점, (b) 포그라운드 복귀 후 유휴 시** 처리. 백그라운드 추론이 불가피하면 `beginBackgroundTask` 래핑 + **체크포인트형 트랜잭션(부분 완료 안전)** 으로 한정하고, 미완료는 다음 기회 재처리로 멱등 설계.
2. **FTS5 동기화 명시(§1)**: DDL에 AFTER INSERT/UPDATE/DELETE 트리거 3종(`new.rowid` 기준) 추가 — **또는** 더 단순하게 폴백을 `LIKE '%query%'` 스캔으로 대체(수백 건 규모 + 한국어 2자 부분일치 정확). FTS5를 유지하면 트리거 + rowid 역매핑을 §1에 못박을 것.
3. **메모리 tool 경로 분리(M-D11/§2)**: `ToolDefinition.InputSchema`를 다중 속성(`[String: Property]` 또는 임의 JSON schema)으로 **일반화**(현 query-only 깨짐 해소). 그리고 **search_memory는 채팅 tool 루프에서 빼고 M-D9 암묵 주입으로 일원화**(Gemini 권고)하거나, 메모리 tool 전용 별도 경량 경로 + `maxToolRounds` 예산 상향(4~5)으로 분리. §2의 "AnthropicChatClient 변경 없음" 문구 정정.

**[P1 — Major, 구현 중 필수]**
4. **프라이버시 명시(M-D2/M-D7/M-D8)**: DB 파일에 `NSFileProtectionCompleteUntilFirstUserAuthentication` 이상 부여(또는 SQLCipher 도입 여부 결정). HITL 삭제는 `valid_to` 무효화가 아니라 **물리 DELETE + VACUUM**(진짜 삭제 권리). "삭제 금지" 원칙은 *모순 무효화*에만 적용하고 *사용자 명시 삭제*와 구분.
5. **추출 품질 강화(M-D6/§3)**: 지속성·부정문 전용 few-shot 예시("매운 거 못 먹어"=저장 / "매운 거 좋아하냐고 물어봄"=폐기)를 추출 프롬프트에 박고, **M0 미니 게이트에서 정확도 정량 측정**(임계 미달 시 보강). 초기엔 LLM 모호판정 제거하고 append-only로 시작.
6. **점수식 잠정 초기값(M-D9/§8)**: α/β/γ·θ에 잠정값 명시(예: α=0.5, β=0.3, γ=0.2; θ_same≈0.82, θ_topic≈0.60) — M0 데이터로 보정. 무명시 상태로 구현 진입 금지.

## ⑤ 구현 시 결정 권고

**GRDB vs raw sqlite3 (§8 열린 결정):**
- **raw sqlite3(`import SQLite3`) 권고.** 근거: (1) .xcodeproj 수정 불필요(SPM 의존성 추가 리스크 회피 — 제약에 명시됨), (2) FTS5/트리거/`LIKE`/BLOB 코사인은 모두 raw SQL로 충분, (3) 1인·greenfield라 GRDB의 타입 안전·마이그레이션 편의보다 의존성 0의 가치가 큼. **단** raw sqlite3는 C 포인터·수동 바인딩·트랜잭션 관리가 번거로움 → **얇은 Swift 래퍼(MemoryStore) 한 겹**으로 prepared statement·트랜잭션·BLOB 인코딩을 캡슐화할 것. FTS5 트리거를 쓸지 LIKE로 갈지가 raw의 복잡도를 좌우 — **LIKE 폴백을 택하면 raw가 한결 단순해져 raw 선택을 더 강하게 지지.**

**폴백 검색 방식:** v0.2 규모(수백 건)에선 **`LIKE '%query%'` 스캔**을 1순위로(한국어 2자 부분일치 정확·트리거 불요·구현 최소). 회상량이 커지면 trigram FTS5(+최소 3자 가드)로 격상.

**임베딩:** Qwen3-Embedding-0.6B Q8_0 + `--embeddings --pooling last` + 쿼리측 instruct 프리픽스 — §8 주의사항 양호. **M0 한국어 유사도 새니티가 검증할 것**(이게 실패하면 M3 전체를 FTS/LIKE-only로 다운스코프).

**추출/주입 위치:** 암묵(배치)=newChat/유휴 경계, 명시 회상=M-D9 암묵 주입으로 흡수(search_memory tool 제거 권고). save_memory만 명시 tool로 남기되 스키마 일반화 후.

## ⑥ 외부 리뷰 통합 / 스킵

- **Gemini(agy): 성공.** Blocker 3(ScenePhase·FTS5 sync·ToolDefinition), Major 5, Minor 3. 코드 직접 검증 양호. **단 "UUID PK가 rowid 매핑을 깸"(Major)은 실측으로 과장 판정 → Minor로 하향**(③-D). trigram 완화책도 2자 한국어엔 불완전(실측 정정).

- **환각 필터 결과**: 외부 리뷰가 단정한 코드 사실(ScenePhase 부재, ToolDefinition query-only, serve.sh 단일 인스턴스, FTS 동기화)을 전부 직접 재확인 — 환각 없음. FTS5 동작·trigram 최소길이·rowid 보존은 내부가 SQLite로 실행 검증.
